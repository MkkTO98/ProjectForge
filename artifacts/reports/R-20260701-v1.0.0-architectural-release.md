# ProjectForge v1.0.0 Architectural Release

Date: 2026-07-01
Version: ProjectForge v1.0.0
Status: Architecturally Stable
Architecture: Frozen, subject to constitutional evidence-based evolution

## Release statement

ProjectForge v1.0.0 is the first stable architectural operating system baseline for AI-native, long-lived software projects.

The release contains five stable architectural systems:

1. Project Identity
2. Context and Continuity
3. Governance and Decision
4. Work Execution Methodology
5. Validation and Evidence

These systems are stable architectural infrastructure. Future ProjectForge development should be incremental, subsystem-local, and evidence-driven. No sixth architectural subsystem should be introduced without satisfying the constitutional evidence standard.

## Architectural achievements

### Relocation from EIP

ProjectForge is no longer treated as an EIP project. It is a general project initialization and governance framework that can be used by any collection of projects. EIP is now one consumer ecosystem rather than ProjectForge's owning context.

Relocation provenance is preserved in `artifacts/reports/R-20260618-eip-relocation-retrospective.md`.

### Generated-project independence

Generated projects are autonomous after creation. ProjectForge improves itself and future inheritance, but does not own generated-project goals, architecture, implementation, task execution, or adoption decisions.

Future ProjectForge improvements do not implicitly modify existing generated projects. Adoption is always an explicit generated-project decision.

### Elimination of runtime coupling

ProjectForge is a framework and scaffold renderer, not a runtime controller for instantiated projects. Generated projects receive local files, tools, state, governance artifacts, and validation expectations; they do not depend on a ProjectForge runtime service.

ProjectForge no longer hosts an embedded MetaHarvest fallback project tree. MetaHarvest is an external advisory provider when configured and remains advisory only.

### Canonization process

ProjectForge v1 matured through bounded canonization of proven architectural responsibilities. The canonization process avoided speculative subsystems and promoted only responsibilities that had become durable architecture:

- identity and authority boundaries;
- context and continuity discipline;
- governance and decision records;
- work execution methodology;
- validation and evidence.

ProjectForge canonizes patterns, not examples, projects, or domains.

### Five-system architecture implemented

The v1 architecture consists of:

1. Project Identity — purpose, scope, non-scope, instruction hierarchy, responsibility boundaries, and independence doctrine.
2. Context and Continuity — bounded startup context, current-state pointers, recovery, latest handoff, context bundles, and closeout discipline.
3. Governance and Decision — durable task, decision, report, handoff, approval, and architecture-audit records.
4. Work Execution Methodology — bounded implementation slices, implementation restraint, verification expectations, and evidence-gated architecture evolution.
5. Validation and Evidence — deterministic structural validation, project-local invariant validation, architectural drift detection, context boundedness validation, generated-project independence validation, evidence preservation, and validator ownership boundaries.

### Constitutional architecture freeze

`CONSTITUTION.md` now declares the five-system v1 architecture stable infrastructure. Future architectural work should assume these systems remain the permanent foundation unless implementation evidence proves otherwise.

### Evidence-gated future evolution

A sixth subsystem may be created only when repeated implementation evidence from multiple independent generated projects demonstrates that no existing subsystem can reasonably own the responsibility. Architectural elegance, theoretical completeness, abstraction opportunities, or anticipated reuse are insufficient evidence.

## Architectural principles

### Project independence

Generated projects remain autonomous after creation. ProjectForge has no ongoing architectural authority over generated projects and does not impose future upgrades.

### Evidence-gated evolution

Architecture changes require implementation evidence. New subsystems require repeated evidence across multiple independent generated projects.

### Canonization doctrine

Reusable capabilities should first emerge independently in generated projects. Only after repeated convergence should they be promoted into ProjectForge.

### Responsibility ownership

Every ProjectForge artifact, tool, template, policy, and document should belong clearly to exactly one of the five systems. If ownership cannot be identified, the artifact is misplaced, the subsystem boundaries need evidence-backed refinement, or the artifact should not exist.

### Bounded architectural growth

ProjectForge should refine existing systems before adding new structure. Shared miscellaneous infrastructure is discouraged.

### Minimalism

ProjectForge should remain intentionally small. Adding reusable infrastructure carries ongoing maintenance cost, so the burden of proof lies with expansion. When uncertain, leave a capability inside an individual project until repeated evidence demonstrates broader reuse.

## Compatibility

- Projects generated before v1 remain supported where practical.
- Future projects inherit the v1 architecture by default.
- Generated projects remain autonomous.
- ProjectForge does not impose future upgrades.
- Existing generated projects may adopt v1 improvements through their own project-local governance.
- Future ProjectForge changes affect future inheritance unless an existing project explicitly chooses adoption.

## Known intentional limitations

ProjectForge is intentionally not:

- an orchestration platform;
- a CI/CD framework;
- a knowledge graph;
- a project-management system;
- an issue tracker;
- a workflow engine;
- a quality-management system;
- a generic static-analysis framework;
- a runtime controller for generated projects;
- an ecosystem owner or cross-project authority.

ProjectForge intentionally avoids speculative infrastructure. It prefers file-backed, understandable, auditable, locally verifiable project systems.

## Future development policy

Future ProjectForge development should consist of:

- incremental refinement;
- implementation-driven canonization;
- subsystem-local improvements;
- evidence-backed architectural evolution;
- maintenance of generated-project independence;
- preservation of concise current-state files and durable provenance artifacts.

No sixth architectural subsystem should be introduced without satisfying the constitutional evidence standard.

## Verification baseline
