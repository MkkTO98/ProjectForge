# Soft Governance Risk Future Review Note

Date: 2026-06-15
Status: recorded for future review only
Scope: risk preservation; no controls implemented

## Question

At what point do repeated high-confidence recommendations begin functioning as de facto governance?

## Context

MetaHarvest is advisory-only. It may preserve reusable non-domain knowledge and generate recommendations, but it must not govern projects, enforce standards, create tasks inside projects, automatically adopt recommendations, automatically implement changes, or modify target projects.

However, repeated acceptance of high-confidence recommendations can create implicit authority even when formal doctrine says recommendations are advisory.

## Risk

A recommendation system can become soft governance if projects begin treating its outputs as defaults that require justification to reject.

This risk is especially relevant when recommendations include:

- high confidence scores;
- repeated successful prior outcomes;
- ecosystem-wide relevance;
- governance-pattern recommendations;
- interface-contract suggestions;
- commit-boundary or sequencing advice;
- project-creation or extraction recommendations.

## Examples

- A project repeatedly accepts MetaHarvest interface-pattern recommendations until future interface choices are treated as pre-decided.
- High-confidence governance-pattern recommendations become effective policy without project-local doctrine adoption.
- Commit-boundary recommendations become treated as mandatory staging instructions.
- Vocabulary recommendations become de facto standards even though terminology adoption was supposed to remain local.
- Recommendation rejection becomes socially or procedurally costly despite formal autonomy.

## Future governance review opportunity

Future review should examine whether the ecosystem needs explicit safeguards distinguishing:

- advice;
- defaults;
- norms;
- policy;
- governance authority.

Potential questions:

- Should high-confidence recommendations require explicit local acceptance before becoming defaults?
- Should recommendation artifacts include stronger advisory-only boundary flags?
- Should repeated acceptance be tracked as evidence, not authority?
- Should projects periodically review whether recommendations have become implicit doctrine?
- How can rejection remain legitimate and low-friction?

## Explicit non-goals

This note does not solve the problem and does not implement controls.

It does not authorize:

- new governance authority;
- new mandatory standards;
- recommendation enforcement;
- automatic adoption controls;
- interface implementation;
- project restructuring;
- task creation.

## Boundary

The question is preserved for future consideration only.
